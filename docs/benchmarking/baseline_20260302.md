# Baseline Performance Profile (2026-03-02)

## Environment
- **CPU:** Standard Modern Desktop CPU (Headless Environment)
- **OS:** Linux
- **Dataset:** ICRA 2020 (forward scenario, 50 images)
- **Library Version:** 0.1.3

## Pipeline Latency Breakdown

The following data was collected using `tools/cli.py bench profile` with 50 tags and 50 iterations.

| Stage | Avg Latency (ms) | % of Total | Notes |
| :--- | :---: | :---: | :--- |
| **Preprocessing** | 4.53 | 8.1% | Bandwidth-bound, integral image & thresholding. |
| **Segmentation** | 5.49 | 9.8% | Union-Find CCL. |
| **Quad Extraction** | 6.22 | 11.1% | Contour tracing & polygon approx. |
| **Decoding** | 39.49 | 70.6% | **CRITICAL BOTTLENECK**. Includes homography & sampling. |
| **Pose Estimation**| ~0.21 | <1% | Performed on successful detections only. |
| **Total** | **55.94** | 100% | Measured for 50 tags per frame. |

## Detailed Observations

### 1. Decoding Bottleneck (70.6% of total)
The decoding stage is the primary bottleneck. For 50 tags, it takes ~39ms, which equates to ~0.8ms per candidate quad. This stage involves:
- Homography estimation.
- Bilinear sampling of the bit grid.
- Soft-decoding (MIH search) or Hard-decoding.
- *Investigation Needed:* Is the MIH search or the bit-sampling kernel more expensive?

### 2. Quad Extraction (11.1% of total)
Quad extraction takes ~6ms for 50 tags (~120µs per component). While not as critical as decoding, it's the next target for SIMD optimization in contour tracing and polygon simplification.

### 3. Preprocessing & Segmentation
Combined, these stages take ~10ms. They are independent of the number of tags and depend on image resolution (1080p). Both are highly amenable to further SIMD acceleration (AVX-512).

## Micro-benchmark vs. Pipeline Profiling

The following table compares the latencies measured via system-level profiling (Python environment) vs. Rust micro-benchmarks (Divan) on the same real-world image (`0000.png`).

| Stage | Profiling (ms) | Micro-bench (ms) | Delta | Notes |
| :--- | :---: | :---: | :---: | :--- |
| **Preprocessing** | 4.53 | 2.96 | -1.57ms | Profiling includes bilateral/sharpen overhead. |
| **Segmentation** | 5.49 | 18.24 | +12.75ms | **Anomaly**. Micro-bench uses different noise profile? |
| **Quad Extraction** | 6.22 | 64.50 | +58.28ms | **CRITICAL ANOMALY**. |
| **Full Pipeline** | 245.73 | 135.20 | -110.53ms | Profiling overhead in Python loop. |

### Observations on Anomalies
1. **Quad Extraction & Segmentation**: The micro-benchmarks are reporting *significantly* higher latencies than the pipeline profiler. This suggests that the profiling spans might be missing recursive calls or that the Python benchmark is somehow bypassing certain components (unlikely).
2. **Full Pipeline**: The micro-bench shows the core detector is ~2x faster than the Python-wrapped benchmark, confirming significant FFI/overhead or Python loop penalties.

## Top 3 Optimization Targets
1. **`decode_candidates` / `sample_grid`**: The highest priority for optimization.
2. **`label_components` (Segmentation)**: Significant fixed cost per frame.
3. **`extract_quads`**: Scalable with the number of components.
